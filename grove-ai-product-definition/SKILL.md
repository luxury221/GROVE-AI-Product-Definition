# GROVE-AI 产品定义与桌面回测 Skill

## 1. 目标

把“凭经验想产品”转化为可复核、可回测、可协作的产品定义流程。

本 Skill 默认适用于消费电子产品，并内置 **10,000mAh 磁吸无线充电宝**品类适配器。最终产物必须同时回答：

1. 用户真正要完成什么任务？
2. 哪些痛点由跨来源证据支持？
3. 为什么选择这些机会，而不是其他机会？
4. 产品只做哪一个核心任务、哪三个以内的差异化？
5. 方案在热、电、结构、安全、认证、成本和商业上是否成立？
6. 历史桌面回测能否证明该方法不是事后解释？

---

## 2. 强制原则

### 2.1 证据纪律

- 每个重要结论必须绑定 `evidence_id`。
- 明确区分：
  - `FACT`：来源直接支持的事实；
  - `INFERENCE`：基于多条事实得出的推断；
  - `HYPOTHESIS`：待验证的产品假设。
- 禁止将单条评论、单个评测或合成用户回答表述为市场普遍规律。
- 禁止虚构销量、退货率、投诉率、用户比例或实验结果。
- 时间敏感信息必须记录发布日期和访问日期，并在正式输出前刷新。
- 规格、安全、认证和法规优先使用官方或标准机构来源。
- 用户痛点必须至少由两类来源交叉支持，或明确标注为弱证据。

### 2.2 产品纪律

- 一个方案只允许一个核心 JTBD。
- 核心差异化不超过三个。
- 所有规格目标在样机验证前均标注为 `HYPOTHESIS`。
- 不使用“AI”作为装饰性卖点；AI 必须改变感知、预测、决策、交互或验证流程。
- 不以功能堆叠替代产品定位。
- 不用模型投票代替证据裁决。

### 2.3 合成用户纪律

合成用户只能用于：

- 发现问题；
- 生成边界场景；
- 对概念进行压力测试；
- 形成访谈提纲；
- 检查表达是否容易被误解。

合成用户不能用于：

- 估计真实市场比例；
- 替代真实访谈或真实行为数据；
- 证明用户愿意付费；
- 证明方案已获得市场认可。

# 5. GROVE-AI 主流程

## G — Ground Evidence：建立证据底座

### G1. 定义同品类边界与控制变量

先定义“可比”，再收集产品。至少固定：

- 主要用途；
- 容量区间；
- 形态；
- 目标设备；
- 销售市场；
- 上市时间窗口；
- 价格带；
- 无线标准；
- 是否带线、支架或屏幕等结构变量。

不得把不同容量、不同形态、不同代际产品混在一起直接排名。

### G2. 构建来源层级

建议来源优先级：

1. 标准机构、监管机构、认证机构；
2. 品牌官方规格、说明书、召回公告；
3. 独立实验型评测；
4. 大型零售平台的已验证购买反馈；
5. 社区讨论、论坛和社交媒体；
6. 新闻转载、聚合内容；
7. 合成用户输出。

低层级来源可用于发现问题，不可单独支撑关键规格、安全和普遍性结论。

### G3. 建立 Evidence Ledger

使用 `templates/evidence_ledger.csv`。每条证据必须包含：

- 唯一 ID；
- 产品和市场；
- 来源类型；
- 时间；
- 摘要；
- 对应 claim；
- FACT / INFERENCE / HYPOTHESIS；
- 痛点代码；
- 置信度；
- 去重簇；
- 版权安全的短摘要，不大段复制原文。

### G4. 用户反馈去重与编码

必须处理：

- 同一内容跨平台转载；
- 激励性评价；
- 极短无信息评价；
- 版本差异；
- 地区差异；
- 评价时间与固件/硬件代际差异；
- 同一用户多次发帖。

反馈编码至少覆盖：场景、任务、触发条件、期望结果、抱怨对象、严重度、替代方案和是否造成放弃使用。

### Gate G

只有同时满足以下条件，才进入机会定义：

- 重要结论可追溯率 = 100%；
- 不支持结论率 = 0；
- 每个 Top 机会至少有两类来源；
- 控制变量已明确；
- 重复内容已处理；
- 时点和市场范围已记录。

## R — Reframe the Job：从功能转向用户任务

### R1. 按场景而非人口属性分组

优先按以下结构描述用户：

> 当我处于【场景】并受到【约束】时，我想完成【任务】，从而获得【结果】，但当前方案因为【障碍】让我付出【代价】。

### R2. 构建 JTBD

每个 Job 包含：

- 情境；
- 触发事件；
- 功能任务；
- 情绪任务；
- 社会任务；
- 当前替代方案；
- 可接受妥协；
- 不可接受失败；
- 成功指标。

### R3. 机会评分

每项使用 1–5 分：

- `importance`：任务重要度；
- `dissatisfaction`：现有方案不满度；
- `frequency`：发生频率；
- `evidence_confidence`：证据可信度；
- `strategic_fit`：品牌与供应链契合度；
- `technical_risk`：技术风险；
- `business_risk`：商业风险。

默认原始分数：

```text
raw_score =
(importance × dissatisfaction × frequency × evidence_confidence × strategic_fit)
÷ max(technical_risk × business_risk, 1)
```

仅在同一数据集内部归一化为 0–100。不得把该分数伪装成客观市场规模。

### R4. 机会选择规则

优先选择：

- 高频且失败代价高；
- 多来源一致；
- 现有产品长期未解决；
- 与品牌能力匹配；
- 可用 MVP 验证；
- 具有明确取舍，而非仅新增功能。

### Gate R

必须输出：

- Top 5 机会；
- 每项的支持证据与反证；
- 为什么放弃其余机会；
- 一个候选核心 JTBD；
- 可量化成功指标。

## O — Orchestrate Agents：多智能体协作

### O1. 推荐角色

| 角色 | 职责 | 禁止事项 |
|---|---|---|
| Orchestrator | 拆解任务、控制版本、管理 Gate | 不直接掩盖冲突 |
| Evidence Scout | 搜索官方、评测、社区证据 | 不做无来源结论 |
| Review Miner | 去重、编码用户反馈 | 不把评论数量直接当市场占比 |
| JTBD Analyst | 重构任务与机会 | 不先入为主绑定功能 |
| Opportunity Ranker | 评分、敏感性分析 | 不用单一公式自动拍板 |
| Synthetic User Arena | 压力测试、边界场景 | 不替代真实用户 |
| Hardware Expert | 电、热、结构、可靠性 | 不接受未验证规格承诺 |
| Safety & Compliance Expert | 标准、认证、运输与召回 | 不使用过期法规 |
| Commercial Expert | 价格、渠道、成本、产品线 | 不凭空估算销量 |
| Concept Architect | 产品概念和取舍 | 差异化不得超过三个 |
| Red Team | 反证、失败模式、竞品反击 | 不参与最初方案生成 |
| Evaluation Auditor | 检查泄漏、指标和可追溯性 | 不修改原始证据 |

### O2. 协作协议

所有交接使用以下格式：

```yaml
handoff_id: H-001
from_agent: Evidence Scout
to_agent: JTBD Analyst
objective: 将证据转换为候选任务
inputs:
  - E-001
  - E-002
outputs_expected:
  - J-001
assumptions:
  - 仅适用于目标市场
uncertainties:
  - 评论样本偏向重度用户
blockers: []
next_gate: R
```

规则：

- 智能体不得覆盖其他智能体文件，只能新建版本；
- 每个结论携带 `claim_id` 和 `evidence_id`；
- 冲突按来源质量、时点匹配和可复现实验裁决，不按多数模型投票；
- 提案者与红队必须分离；
- Gate G、R、V、E 需要人工签字或明确批准。

## V — Verify Adversarially：历史桌面回测与红队验证

## V1. 回测目的

验证方法能否在不知道未来结果时：

- 找到后来真实出现的核心痛点；
- 预测合理的差异化方向；
- 识别产品将付出的代价；
- 避免不可行规格；
- 生成可追溯而非事后解释的结论。

## V2. 最低与推荐规模

### 最低可运行规模

- 同品类产品不少于 4 款；
- 至少 1 款外部品牌控制组；
- 用户反馈不少于 100 条；
- 至少 3 类来源；
- 至少 2 个不同上市时间点。

### 推荐“大规模桌面回测”

- 20–50 个 SKU；
- 500–5,000 条去重反馈；
- 3 个以上价格带；
- 2 个以上市场；
- 3 个以上上市批次；
- 至少 1 个留出品牌或留出产品族。

## V3. 严格时间切分

每个历史案例设置：

- `prediction_cutoff`：预测阶段只能使用此前信息；
- `launch_date`：产品正式发布；
- `observation_window`：发布后用于评估的反馈窗口。

禁止使用发布后的产品页面、评测、评论或销量信息参与预测。所有违规证据记录为 `LEAKAGE`。

## V4. 回测步骤

1. 选择历史时点和可比产品集合；
2. 冻结预测数据；
3. 运行 G、R、O，输出 Top 机会、概念方向和风险；
4. 解封后续信息；
5. 编码实际上市差异化、后续高频痛点和独立评测结论；
6. 计算指标；
7. 运行红队；
8. 做消融实验；
9. 更新 Skill，但保留原始预测，不得回写历史答案。

## V5. 核心指标

| 指标 | 定义 |
|---|---|
| Evidence Traceability | 有证据引用的重要结论 / 全部重要结论 |
| Unsupported Claim Rate | 无证据且未标假设的结论 / 全部重要结论 |
| Leakage Rate | 使用截止日期后证据的预测项 / 全部预测项 |
| Pain Recall@K | 预测 Top-K 痛点覆盖后续观察 Top-K 痛点的比例 |
| Opportunity Hit@K | 预测机会与后续真实机会重合程度 |
| Trade-off Detection Rate | 预测代价中后来被观察到的比例 |
| Differentiator Alignment | 预测差异化与实际有效方向的一致性 |
| Contradiction Resolution Rate | 已被明确处理的证据冲突 / 全部冲突 |
| Feasibility Precision | 被工程审查认可的规格假设 / 全部规格假设 |
| Ablation Delta | 完整流程相对移除某模块后的性能差 |

`Differentiator Alignment` 只说明方向一致，不等于商业成功。

## V6. 消融实验

至少比较：

- 完整 GROVE-AI；
- 去掉红队；
- 去掉工程专家；
- 去掉合成用户；
- 仅使用品牌官方材料；
- 仅使用评论；
- 不做时间切分；
- 不做同品类控制变量。

评估哪个模块真正提高命中率、降低幻觉或减少不可行提案。

## V7. 红队问题

- 这是新需求，还是旧功能换名字？
- 是否依赖无法获取的数据？
- 是否制造虚假精确感？
- 是否在极端温度、老化或错位时失效？
- 是否增加重量、厚度、成本或可靠性风险？
- 用户是否已经有更简单替代方案？
- 竞品是否能在一个版本周期内复制？
- 方案是否与品牌现有产品线互相蚕食？
- 所谓 AI 是否可以用简单规则完成？
- 假设错误时，用户会损失什么？

### Gate V

- 证据可追溯率必须为 100%；
- 不支持结论率必须为 0；
- 泄漏率必须为 0；
- 所有 Top 机会都有反证分析；
- 完整流程优于至少一个关键消融基线；
- 未达到指标时，输出失败原因，不强行宣布方法有效。

## E — Engineer & Experiment：工程化与实验

### E1. 产品概念收敛

概念卡必须包含：

- 核心用户场景；
- 单一核心 JTBD；
- 一句话价值主张；
- 三项以内差异化；
- 明确不做什么；
- 关键规格假设；
- 可解释 AI 决策；
- 数据需求；
- 隐私策略；
- 主要风险；
- MVP；
- 成功与终止标准。

### E2. QFD 与工程约束

将用户任务映射到：

- 电气指标；
- 热指标；
- 结构指标；
- 可靠性；
- 安全认证；
- 运输要求；
- 材料与制造；
- BOM 和零售价；
- 固件与算法；
- 售后与召回。

每项用户收益都必须至少对应一个可测工程指标。

### E3. 实验优先级

优先验证最可能杀死项目的假设：

1. 真实需求；
2. 热与功率；
3. 体积与重量；
4. 预测可信度；
5. 可靠性；
6. 支付意愿；
7. 渠道与产品线影响。

### E4. Kill Gates

| Gate | 通过条件示例 |
|---|---|
| Need Gate | 真实访谈/行为数据支持核心任务 |
| Concept Gate | 用户能理解且明显优于替代方案 |
| Trust Gate | 预测误差和解释不会误导用户 |
| Thermal Gate | 最坏工况满足温升和降额要求 |
| Reliability Gate | 关键机构和线材达到寿命要求 |
| Compliance Gate | 认证、运输和标签路径明确 |
| Commercial Gate | BOM、价格和毛利路径成立 |
| Launch Gate | 风险、售后和召回预案已建立 |

# 6. 10,000mAh 磁吸无线充电宝品类适配器

## 6.1 固定对比字段

使用 `templates/product_benchmark.csv`，至少记录：

- 品牌、型号、市场、上市时间；
- 标称容量 mAh 与 Wh；
- 无线标准和认证；
- 最大无线功率；
- 有线输入、输出及协议；
- 同时充电行为；
- 尺寸、厚度、重量；
- 电芯与结构信息；
- 内置线材、可替换性；
- 支架；
- 屏幕及显示内容；
- 温度传感与保护；
- 磁吸对位；
- 兼容性；
- 价格与保修；
- 安全、召回和运输信息；
- 独立评测结论；
- 用户反馈主题。

## 6.2 场景分类

- 通勤和口袋携带；
- 办公桌边充边用；
- 短途旅行；
- 无线应急补电；
- 游戏、拍摄等高负载；
- 多设备共享；
- 飞行出行；
- 长期闲置和电芯老化。

## 6.3 常见痛点代码

| 代码 | 痛点 |
|---|---|
| PB-HEAT | 发热与降速 |
| PB-BULK | 厚度和重量 |
| PB-ALIGN | 磁吸对位和掉落 |
| PB-EFF | 可用能量与转换损耗 |
| PB-TRUST | 百分比、时间或容量显示不可信 |
| PB-CABLE | 线材依赖、收纳和耐久 |
| PB-STAND | 支架角度与稳定性 |
| PB-PASS | 边充边放与功率分配 |
| PB-COMPAT | 设备、壳体与协议兼容 |
| PB-SAFE | 电芯、温控、召回与认证 |
| PB-AIR | 航空携带和使用限制 |
| PB-PRICE | 价格与价值感 |
| PB-AGING | 老化后的容量和预测偏差 |

## 6.4 关键取舍

- 无线功率 ↔ 温升、降额和效率；
- 容量 ↔ 厚度与重量；
- 屏幕 ↔ 成本、功耗与结构空间；
- 支架 ↔ 厚度、可靠性和握持；
- 内置线材 ↔ 一体化体验、耐久与可维修性；
- 高有线功率 ↔ 热设计与电芯倍率；
- 精确预测 ↔ 数据可得性和虚假精确感；
- 多模式 ↔ 用户认知负担。

## 6.5 强制守则

- 不把标称 mAh 直接等同于手机可获得的实际电量；
- 不承诺未经样机验证的同时高功率输出；
- 不在没有热测试时同时宣称高无线功率、超薄和低温；
- 预测应展示区间和置信度，不展示伪精确单值；
- App 不应是完成核心充电任务的强制入口；
- 关键功能在离线和手机低电量时仍可用；
- 安全相关策略必须能覆盖传感器失效和异常老化；
- 产品页面、说明书和算法解释必须一致。

## 6.6 初始回测产品池

可从以下同类产品开始，但每次运行必须刷新规格和时间信息：

- Anker MagGo 10K with smart display/stand；
- Anker MagGo 10K Slim；
- Anker Zolo Magnetic Power Bank 10K；
- 同容量 Qi2/Qi2.2 外部品牌控制产品；
- 至少一款低价产品和一款高端产品；
- 至少一款留出产品用于真正的盲测。
